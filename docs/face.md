### doc 知识库 RAG 平台

他是一个代码项目维度的doc知识库，核心就是通过mcp工具给每个项目仓库生成一份文档，然后数据入库存入到向量数据库内，用户可以在ide内通过mcp服务跨项目的了解其他项目内容，并且在每次合并代码的时候触发文档的更新，持续的更新写入文件，自动化的维护文档的内容。
核心目的是需要一份这样的数据去支持做更多的事，ai的发展是很快，在编码方面每个月都有提升，很多你之前做的辅助ai编码，提升编码质量的东西其实都不需要了，包括之前也看到一篇文章我们做的一些harness工程化的东西如何在ai提升的情况下优雅的剔除，如何做不会影响提升以后ai的能力。这些都说明，在一些方面作为业务开发者，没必要去做过度的设计，提升编码，只需要等待就好了，相信ai的能力，我们要做的是即使ai提升了也需要我们辅助才能做到事情，而这个数据的整理，我觉得就是ai提升也无法覆盖解决的问题，他始终需要我们最新的数据文本做为上下文来处理。我个人觉得这个方向是正确的，先把数据整理好一定是第一位，有了这些数据之后其实除了我刚刚说的那些辅助跨项目跨团队的开发以外，也可以以此数据为基础做一些b端项目的agent开发，比如b端很多表格图标之类的，内容，现在都是人去找功能，我觉得在ai时代下不应该还是人去找功能，而是直接想要什么数据直接通过ai梳理展示出来，有了那份数据之后，我就可以在项目各个域上去实现这个内容，我们已经有了这个域内的所有数据，接口信息，返回数据，展示的数据等，让后就可以让ai去请求用户需要的数据，重新的梳理展示出来，包括还有我们的活动页配置，低代码的组件配置，其实就是json的配置，这个也可以让ai很好的来实现，我们有活动平台的数据等等就可以很好的实现。
当然关于活动页配置是否用ai来实现，我其实是比较纠结的，因为我觉得如果还是用ai来低代码的配置json，本质还是没有脱离原来的低代码的结构，无非是交互方式的改变从手动拖动改成语言描述的生成，生产力是否有提升其实是有一个很大的提升的，最近看了一个工具用html页面代表ppt给了我一个很好的启发，其实html是比ppt更好的一种文档表现形势，但是碍于html的生产是有门槛，且比较费时，所以ppt是更好的一种表现形式，但是在ai时代下，这种门槛和费时完全可以被忽略，就很多我们觉得复杂或者费时的优化，其实在ai时代是可以重新梳理推翻的，比如活动页，我们之前是低代码的方式去配置页面因为每次活动页都去单独配置的话是十分费时的，但是在ai时代下，活动页的生成是否就可以改造成agent的实时生成，配置一些模版，接入claude design,直接生成活动页，配置一些基础的登陆等等就好了，活动页的核心其实就是画页面，功能大部分其实并不复杂，我觉得是一个可以探索的方向。
然后知识库整个的架构层面的话，
首先我们是mcp-server的形式去提供工具给到每个用户，而不是平台化的形式，因为我理解ai时代内部应该是一个去平台化的过程，开发者的平台只有一个个人的IDE编辑器，或者cc这种的终端IDE形式，不需要去做平台，想要什么功能自己接入就好，我们把整个流程拆成多个工具；
首先是冷启动，直接借助ide内置的llm去阅读整个项目，生成两层架构的信息内容，第一层是wiki域，把整个项目切分为多个模块，每个模块都有一个详细的wiki来介绍这个模块的内容和对应的文件等等内容，然后第二层是file_doc文件则是读取所有的文件，每个文件，js css等等都会读取，然后会生成一个json，来描述这个文件的内容，很多字段有些字段，这个文件的描述，方法名路径等等，还有描述的向量等，这是我们初识话的内容

然后是向量检索方向上，我们也会做两层检索，先从wiki域上做向量检索，获取到高相似度对应的业务板块，然后从这里获取到文件文档，在去找file_docs里面的对应的json，会把这两部都给到llm，llm会有全局视角和精确的文件视角来辅助回答

这个项目是如何做切块的，首先和按常规的文字数量切块不太一样，我们一个wiki，一个模块wiki就是一个向量模块，然后一个文件json的描述又是一个向量模块，他们两个之间会有一个relation的字段来建立联系，通过此可以找到对应域的所以文件

如果没有找到对应域，我们也会有兜底策略直接filedoc的向量检索，同时向上获取到对应的域，
两者同时发生，然后会打分，重复的权重家高，然后排序，将对应的内容返回过去

如果一个域对应的文件过多，那么首先我们会对其关联的文件做一个二次向量的检索的排序，同时结合之前file——doc的直接查找打分，做一个排名取前10个文件的内容，和前三个域相关的内容，还有就是在大域里面继续拆成小域

当然除此之外之外的话也做了一些常规的优化召回方案的增强，1.改写用户的问题，扩写问题2.混合并发检索3.关联的作用域也加入进来
4.llm精排打分

最后就是我们没次都会更新代码，但是冷启动之后我们代码会更新，我们需要的是一个事实可更新的文档，llm wiki其实是给了我一个思路，在每次代码更新提交mr的时候触发的我们的脚本执行，去读取本次的更新内容，然后ai理解完以后实时的更新到我们向量数据库里面去

除此之外我们也是用graph rag的思想在李敏啊，我们有依赖关系案图表，检索到对应文章然后将依赖文件也加入进来，然后打分可能会出现依赖的依赖也需要这时候我们需要这个打分机制来判断，多跳一层继续减分数，知道不需要，当然我们也可以直接跳一层直接就给到llm做判断，让他来判断是否还需要跳，获取足够的上下文，去哪些文件下继续往下跳。

### 提示词工程平台

设计并搭建统一提示词管理平台，将散落各业务模块的 AI 提示词集中管控，支持版本管理与灰度发布

各业务模块通过接口动态拉取提示词，实现提示词与代码完全解耦，支持热更新与 A/B 实验，无需发版即可生效

将提示词迭代周期从"改代码 → CR → 发版"压缩为"平台修改 → 立即生效"，显著降低 AI 功能的迭代成本

项目就是将各个业务模块的提示词统一放到一个可编辑的提示词平台管理，在这个平台修改提示词以后，就可以实时请求最新的提示词，不用改代码发布，另外性能问题，理论上是可以做缓存的redies 缓存，但是我觉得ai时代其实，用户对于ai的容忍度是比较高，这点优化不影响，并且我们现在的场景大部分是b端对内的场景

### MCP 开发工具集

大模型是没办法接触自身以外的能力，比如操作浏览器或者获取数据库的能力，通过mcp协议，我们就能让ai模型调用外部工具和数据源，就相当于让大模型又了接触外部世界的手和脚。
以 IDE 为例子，ide通过配置项连接mcp server服务以后mcp会列出一系列的工具给到ide客户端，用户提问的时候，ai根据自己判断就去调用对应的工具，传入参数，执行对应的方法获取到数据返回给ai，ai在处理这个数据给到ai
现在使用的是 http 加 sse 的方式启动的

mcp 和 skills的区别本质都是为了服务大模型，但是mcp是工具的调用协议，skills更像是提示词文档的约束歇息，skills是markdown的文本 mcp是可执行的函数，一个是新人入职手册，一个给你开了各种工具控制权限的中心

我的主要是脚手架 或者 公共组件 周报 或者多语言翻译与key的同步。还有一个是打通从后端代码到mapi文档然后mapi文档到前端的整个链路，极大的提高了前后端协同效率。

### 个人博客书单

个人博客书单的话就是上传电子书，然后每个电子书都是一个类似的一个世界吧，我们会把整个小说rag化，然后通过向量搜索去提取人物的个性口头禅等等，然后设置这样的一个人物agent，我们就可以和书中人物对话，或者是扮演书中的人物和另外一个角色对话，这是其中的一部分功能呢。
还有一部分功能的就是直接生产小说，前期ai辅助用户生产大纲，世界观，主要人物塑造，伏笔，文笔风格等等，然后以此为背景上下文来生产小说，会将上下文分成五层结构，刚刚的大纲事件观，文笔风格作为第一层，人物性格动机成长弧作为第二层 然后是章节压缩，当前章节的前20章内容会压缩成阶段摘要，第四层就是最近10章，最近的剧情，第五章用户设定想要完成的本章的情节，和伏笔提醒
以此为基础的创作生产
混合检索，就是稠密检索和稀疏检索，稠密检索就是当前使用的语言检索，用一个几百维或者是几千维的向量来表示语义，比如男人女人这种是性别，他们就用一个维度的向量来表示，最后计算余弦相似度，越靠近的语义越相似，稀疏检索就是篇关键词匹配的方式，一个超大的向量，大部分是0，出现过词的位置是1 ，看次的重叠度重叠越多越相关。

除此之外的还有每章节的分镜生成，我这条支线的最终想法是小说分镜然后再到视频，所以把小说的分镜也尝试去做了，目前可以优先去生成图片素材库，基本的人物和环境的素材库，然后以素材库为背景在加入到当前章节中，图片和章节内容作为提示词去生成分镜，我用的minimax的模型，目前生成效果一般，之后会尝试一下用chatimage2 来生成试试效果。然后在视频环节目前只是初步的做了一些调研，看了一下市场上了，当然也看了一下你们workbuddy的生成，主要就是两条路线，一种是直接调生视频的大模型，太贵了，目前现在这个阶段不太考虑，然后还有一种就是，画html页面，生成png的图片帧，调用一些语音模型minimax，然后用ffmpeg框架去合成视频和语音，目前这部分我是把整个流程跑通了，能生成一些简单的比如财经播报的视频，控制语音和画面同步的方式，主要是依赖返回的语音数据，语音数据除了语音以外还会有语音里面每句话对应的时间，比如一句话面试官你好他会给你一个start 0 end5的毫秒显示，然后下一句话又是多少到多少，然后我们根据返回的语音数据去生图，比如一共我们是25秒的视频，我们定一秒就是一帧的图片所以我们需要生成25张图片，我们去画html 然后用pupeteer 去截图，这样就有对应的25张图，将语音和这些图片输出给ffmpeg 让他以一秒一帧的方式去渲染生成视频就好了，就能生成最后的视频。
