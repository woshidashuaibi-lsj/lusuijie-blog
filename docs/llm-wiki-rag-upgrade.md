# LLM Wiki 模式 × RAG 升级思考

> 原文：《大模型百科》- Andrej Karpathy（Lumina 转载）
> 结合：当前书籍 RAG 对话系统的现状与改进方向

---

## 一、文章核心思想梳理

### 1.1 传统 RAG 的本质问题

当前主流 RAG 的工作方式：

```
用户提问 → 向量检索原始文档 → 拼接相关片段 → LLM 生成答案
```

**问题**：每次提问都从零开始重新检索、重新拼接、重新推导。**没有积累**。

- 需要综合 5 份文档的复杂问题，每次都要重新找、重新拼
- 矛盾信息每次都重新发现，没有被提前标记
- 上次问答产生的洞察，下次问答用不上

### 1.2 LLM Wiki 模式的核心区别

不是"用时检索"，而是"持续编译"：

```
新资料加入 → LLM 主动整合进 Wiki（摘要、交叉引用、更新旧结论）
用户提问   → 查询已整合好的 Wiki → 直接给出带引用的答案
```

**核心区别对比**：

| 维度 | 传统 RAG | LLM Wiki 模式 |
|------|---------|--------------|
| 知识处理时机 | 查询时实时检索 | 资料加入时预先整合 |
| 积累性 | 无，每次从零开始 | 有，Wiki 持续增长 |
| 矛盾处理 | 检索时可能同时返回矛盾内容 | 摄入时已标记矛盾 |
| 交叉引用 | 无 | 已预先建立 |
| 问答结果 | 消失在聊天记录中 | 高价值答案可回写进 Wiki |

**一句话**：RAG 像"临时查资料"，LLM Wiki 像"持续写百科"。

### 1.3 三层架构

```
原始资料层（只读）
    ↓ LLM 摄入
Wiki 层（LLM 生成与维护）
    ↑ Schema 层（规则文档，定义结构与更新规范）
```

- **原始资料层**：文章、论文、图片。不可变，是事实来源
- **Wiki 层**：摘要页、实体页、主题页、对比页、综合结论页。LLM 完全拥有这一层
- **Schema 层**：告诉 LLM 维基结构、命名规范、更新工作流——让 LLM 从聊天助手变成"知识库维护员"

### 1.4 三个操作流程

**摄入（Ingest）**：新资料加入 → LLM 读取并影响 10-15 个 Wiki 页面

**查询（Query）**：先查 Wiki 索引 → 定位相关页面 → 综合给出带引用答案 → 高价值答案回写 Wiki

**巡检（Lint）**：定期检查矛盾、孤立页、过时结论、缺失交叉引用

### 1.5 两个关键文件

- **`index.md`**：所有 Wiki 页面的目录，帮助 LLM 先定位再深入查阅
- **`log.md`**：时间顺序的操作记录（摄入/查询/巡检），追踪 Wiki 演化历史

### 1.6 为什么有效

> LLM 不会感到厌倦，不会忘记更新交叉引用，一次操作就能处理多达 15 个文件。
> 维护成本接近于零，Wiki 得以持续维护。

人类做：**选资料、提问题、做判断**
LLM 做：**摘要、整理、交叉引用、簿记**

## 二、当前书籍 RAG 系统的现状

### 2.1 现有架构

```
用户提问
  ↓
embedQuery(问题) → 阿里云 DashScope text-embedding-v3 → 1024 维向量
  ↓
全量拉取 Supabase rag_documents（WHERE collection = bookSlug）
Node.js 内存中计算余弦相似度 → Top-5 段落
  ↓
组装 System Prompt = 角色人设 + Top-5 段落 + 回答要求
  ↓
MiniMax API → SSE 流式输出 → 前端逐字渲染
```

### 2.2 现有系统的核心限制

**问题 1：每次检索都从原始书籍段落出发**

每次对话，Top-5 段落都是从原始书籍切片中实时检索的，没有任何"上次问了什么、发现了什么"的积累。用户每次换个角度问同一个问题，系统重新检索，可能拿到略有不同的段落，给出略有不同的答案——没有一致性。

**问题 2：书籍内容只被切分，没有被理解**

当前只是按字数切分书籍段落（chunking），没有经过语义理解。一个"哲学隐喻"段落和一个"人物对话"段落在向量空间里可能距离很近，但对回答不同问题的价值完全不同。

**问题 3：向量检索的语义gap**

用户问"李火旺为什么选择成仙"，但原文可能描述的是"他看着星空，心中升起了执念"——字面距离很远，但语义相关。纯向量检索在这种gap上表现不稳定。

**问题 4：角色人设是静态的**

`personas/{bookSlug}.txt` 是预先提取好的静态文件。如果书籍内容更新，人设不会自动修订。如果用户的问答中发现了新的人物特征，也无法被记录下来。

## 三、结合 LLM Wiki 思路的改进方向

### 3.1 核心思路：为每本书建立一个"活的 Wiki"

不只是把书籍内容切片存向量，而是**预先让 LLM 把书籍理解成结构化的 Wiki**，检索时查 Wiki，而非查原始切片。

```
书籍原文（原始资料层，只读）
    ↓ 摄入阶段（一次性 + 增量更新）
书籍 Wiki（中间层）
    ├── characters/李火旺.md   # 人物页：性格、动机、关键决策、成长弧线
    ├── characters/师父.md
    ├── themes/修仙动机.md     # 主题页：书中所有关于"为何修仙"的内容综合
    ├── events/xxx事件.md      # 事件页：关键剧情节点
    ├── concepts/道.md         # 概念页：书中特有的世界观概念
    ├── contradictions.md      # 矛盾标记：前后不一致的设定
    ├── index.md               # 所有页面目录
    └── log.md                 # 摄入/更新记录
    ↓ 查询阶段
用户提问 → 查 Wiki → 精准回答
```

### 3.2 具体改进方案

#### 方案 A：Wiki 预生成 + 向量检索 Wiki 页面（推荐）

**摄入阶段**（书籍加入时做一次）：
```
原始书籍文本
  ↓
LLM 阅读 + 提取
  ↓
生成 Wiki 页面：人物页、主题页、事件页、概念页
存储为 Markdown 文件 + 同时向量化 Wiki 页面
```

**查询阶段**：
```
用户提问 → 向量检索 Wiki 页面（而非原始切片）→ 找到最相关的 Wiki 页面 → LLM 综合回答
```

**优势**：
- 检索的是已经经过 LLM 理解和整合的内容，语义更丰富
- 人物页聚合了书中所有关于该人物的信息，不会遗漏
- 矛盾已预先标记，回答更一致

#### 方案 B：两级检索（Wiki 索引 + 原文引用）

```
用户提问
  ↓
第一级：查 Wiki 索引，定位相关的 Wiki 页面（人物/主题/事件）
  ↓
第二级：从定位到的 Wiki 页面中找原文引用段落
  ↓
LLM 用 Wiki 页面内容组织答案 + 用原文段落提供引用支撑
```

**优势**：既有 Wiki 的结构化知识，又保留了原文引用的可信度。

#### 方案 C：问答结果回写 Wiki（渐进增强）

```
用户提问 → 回答 → 判断是否是高价值洞察
  ↓（如果是）
把这次问答的结论写回对应 Wiki 页面
```

例如：用户问了"李火旺在第 30 章做了什么决定"，LLM 给出了精彩的分析，这个分析可以被写回 `characters/李火旺.md` 的"关键决策"章节。下次再问相关问题时，Wiki 已经包含了这个分析。

### 3.3 Schema 文档设计（对应当前 personas 的升级）

当前的 `personas/{bookSlug}.txt` 只记录角色人设，可以升级为 Schema 文档：

```markdown
# 《道诡异仙》Wiki Schema

## 目录结构
- characters/  人物页（每个主要角色一个文件）
- themes/      主题页（修仙动机、人性、道的本质等）
- events/      关键事件页
- concepts/    世界观概念页

## 人物页格式
- 基本信息（身份、阵营）
- 核心性格（3-5 个关键词 + 原文依据）
- 成长弧线（按章节顺序的变化）
- 关键决策（影响剧情的重要选择 + 动机分析）
- 关联人物（与其他角色的关系）
- 语言风格（对话特征，用于角色扮演）

## 回答规范
- 优先引用 Wiki 页面中的整合内容
- 涉及原文细节时，标注来源章节
- 遇到矛盾时，优先以 contradictions.md 中的标注为准
```

### 3.4 改进后的检索流程（代码层面）

**现在**：
```typescript
// 现在：全量拉取 + 内存计算余弦相似度
const docs = await getAllDocuments(bookSlug)
const similarities = docs.map(doc => cosineSimilarity(queryVector, doc.embedding))
const top5 = getTopK(similarities, 5)
```

**改进后**：
```typescript
// 第一步：查 Wiki 索引，定位相关 Wiki 页面
const wikiIndex = await readWikiIndex(bookSlug)
const relevantPages = await vectorSearch(queryVector, wikiIndex, topK=3)

// 第二步：读取 Wiki 页面内容
const wikiContent = await readWikiPages(relevantPages)

// 第三步：如果需要原文引用，再做第二级检索
const originalQuotes = await vectorSearch(queryVector, originalChunks, topK=3)

// 第四步：组装 Prompt
const prompt = buildPrompt({
  wikiContent,    // 结构化的 Wiki 知识
  originalQuotes, // 原文引用
  userQuestion,
  persona,
})
```

## 四、实施优先级建议

### 短期（可快速落地）

1. **把当前 `personas.txt` 升级为结构化人物 Wiki**：用 LLM 把已有的角色人设扩展成多维度的人物页，包含性格、动机、关键决策、语言特征
2. **为书中核心概念建立概念页**：《道诡异仙》的"道"、"仙"等核心概念，预先整合书中所有相关段落

### 中期（需要一定开发量）

3. **建立书籍 Wiki 目录（index.md）**：让检索先查索引定位，再深入 Wiki 页面，而非直接检索所有原文切片
4. **实现两级检索**：Wiki 页面 + 原文引用，兼顾结构化知识和原文可信度

### 长期（进阶）

5. **问答结果回写机制**：高价值的问答洞察写回 Wiki，让书籍 Wiki 越用越丰富
6. **巡检机制**：定期检查 Wiki 中的矛盾、孤立页、过时内容

---

## 五、一句话总结

> 当前 RAG 是"每次都去书架上找原文段落"，LLM Wiki 模式是"先让 LLM 把书读完写成百科，以后直接查百科"。对书籍对话场景来说，这意味着从"检索原始切片"升级为"检索书籍 Wiki"，回答更结构化、更一致、更有深度。
