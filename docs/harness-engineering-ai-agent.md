# Harness Engineering 的概念引入

今天我们来聊一个最近在AI圈特别火，但很多人还没真正弄懂的词 —— Harness engineering。如果你最近也在做 agent 或者关注 AI 应用的落地，或多或少可能都会遇到这样的问题：为什么同样的模型？别人做出来的 agent 可以连续跑很久，成功率很高，到了自己手里就总是差强人意呢。很多人可能会想是不是模型不够强，是不是提示词没调好，是不是 IG 没调明白。

当然啊这些都有影响。但是啊越来越多的团队最后发现呢真正决定我们的系统能不能稳定地跑起来，往往不是模型本身，而是模型外面那套运行的系统。那这套东西呢现在有了一个统一的名字就叫 harness。大家好，欢迎来到可乐秘密花园。

我是花园老师，为什么想聊这个话题呢？因为年初的时候啊，有个朋友找我，帮他们调一个 agent，他们团队之前已经做了很多努力，换上了最好的旗舰模型，提示词改了上百版，各种参数也调了不少。但是一进真实的场景效果还是不稳定。有的时候很聪明，有的时候又莫名的跑偏，任务的成功率不到 70%。那后来呢我去帮他们看了一下，最后的改动最大的地方反而不是模型，也不是提示词。我的改进点呢在于：

- 任务是怎么拆的
- 状态是怎么管的
- 关键的步骤要怎么校验
- 失败之后要怎么恢复

新版本上线之后呢，还是同样的模型，同样的提示词，成功率拉到了 **95% 以上**。

那当时这个朋友问我，你到底改了什么呢？说实话啊，那个时候我没有一个特别准确的词来形容，直到最近 harness engineering 这个概念越来越火，我才意识到我当时改的这些东西，本质上就是 harness。

所以今天这期视频呢，我想彻底把这个概念跟大家讲清楚，我们主要分三个部分：

1. harness 是怎么一步步演进出来的
2. 一个成熟的 harness 到底包括哪些部分
3. OpenAI、Anthropic 这些公司真实到底是怎么做的

---

## AI 工程的三次中心迁移

过去两年呢，AI 工程其实经历了三次很明显的中心迁移：

```
Prompt Engineering → Context Engineering → Harness Engineering
```

表面上看，好像只是换了几个新的名词。但如果你只是把它理解成术语流行式，那就完全低估它们了。那这三个词呢分别对应了现在 AI 系统发展的三个阶段性问题：

- 模型有没有**听懂**你在说什么？
- 模型有没有拿到**足够而且正确**的信息？
- 模型在真实的执行中**能不能持续地做好**？

你会发现这些问题呢是一层一层往外扩张的。

---

### 第一阶段：Prompt Engineering

在大模型刚火起来的时候呢，大家最直观的感受就是同一个模型，你换一种说法，结果可能差很多。比如你说一句"帮我总结一下这篇文章"，他可能只会给你一个很平的总结。但如果你换一种说法，效果马上就会不一样。所以那个阶段呢，大家都相信一件事情：模型不是不会，而是你没有把问题说明白。

于是大家开始疯狂地研究提示词，什么角色设定、风格约束、Few-shot 示例、分布引导、输出格式等等等等。

那为什么这些东西有效呢？因为大模型本质上是一个**对上下文非常敏感的概率生成系统**：

- 你给他什么身份，他很容易沿着那个身份去回答
- 你给他什么样的样例，他很容易沿着那个范式去补全
- 你强调什么样的约束，它就很容易把那部分当成重点

所以提示词工程的本质不是命令模型，而是**塑造一个局部的概率空间**。那这个阶段最重要的能力不是系统的设计，而是语言的设计。

但提示词工程很快就遇到了天花板，因为很多任务不是你说清楚就行，而是你真的得知道。比如：

- 让模型分析一份公司的内部文档
- 回答一个产品的最新配置
- 按照一套非常长的规范去写代码
- 在多个工具之间完成复杂的任务

这个时候你会发现提示词写得再漂亮，也替代不了事实本身。所以呢：

- 提示词擅长的是：长期任务约束输出、激发模型的已有能力
- 提示词不擅长的是：凭空弥补缺失的知识、管理大量动态的信息、处理长链路任务里的状态

说白了，**提示词解决的是表达的问题，不是信息的问题**。

---

### 第二阶段：Context Engineering

于是第二阶段开始了。当大家还只是做聊天机器人的时候呢，提示词的作用很大，因为任务短、链路短，状态少，很多问题确实靠把话说明白就可以解决了。但后来 agent 开始火了，模型不只是要回答问题，而是要进到真实的环境里面做事情。他要多轮对话、调浏览器、写代码、操作数据库，这些工具还要在多个步骤之间传递中间结果，还要根据外部的反馈不断修订计划。

那这个时候问题就变了，系统面对的已经不是"一次回答对不对"，而是**整条链路的任务能不能跑通**？

比如如果你不是简单的问一句"帮我总结一下这篇文章"，而是让他做一个更真实的任务，比如说"帮我分析这份需求文档，找出潜在风险，结合历史的评审意见，给出改进建议，再生成一版发给产品经理的反馈稿"。你会发现这已经完全不是一句提示词就能解决的问题了。他至少要拿到：

- 当前的需求文档
- 历史的评审记录
- 相关规范
- 当前目标
- 已经分析出来的中间结论
- 输出的对象是谁、语气应该怎么调

所以 context engineering 的核心就变成一句话：**模型未必是知道的，系统必须在合适的时机，把正确的信息送进去。**

那这里的 context 呢也不只是几段背景资料，在工程的意义上，它代表了**所有影响模型当前决策的信息的总和**，包括：

- 用户的输入、历史对话
- 检索结果、工具返回
- 当前任务的状态、中间产物
- 系统规则、安全约束
- 其他 agent 传过来的结构化结果

所以你会看到 prompt 其实只是 context 的一部分。

说到 context engineering 呢，我觉得 **RAG（检索增强生成）** 也算是一个比较典型的实践。RAG 的价值是很直接的：模型参数里面没有知识，怎么在运行时补进去呢？做法大家都知道，先检索再把相关的内容塞到上下文。

但是真正成熟的 context engineering 关注的肯定不只是检索，他关注的是整条完整的链路，比如：

- 文档怎么切块，结果怎么排序？
- 长文怎么压缩？
- 历史对话什么时候要保留，什么时候要摘要？
- 工具返回要不要全部暴露给模型？
- 多个 Agent 之间到底传原文、摘要还是结构化的字段呢？

包括最近很火的 **Agent Skills**，我觉得本质上也是上下文工程的高级实践，因为它解决了一个特别现实的问题：如果你把十几个不同的工具说明、所有的参数定义全部一上来就塞给模型，理论上模型会知道的更多，但是实践往往会更糟糕。为什么呢？因为上下文的窗口是非常稀缺的资源，信息一多注意力就会涣散。

所以 skill 采用的是一个非常典型的思路，叫**渐进式披露**：不是一开始就把能力全部给模型看，而是只给他看最少量的原信息，等他真正要触发某些能力的时候，再把那部分的 SOP、详细的参考信息、脚本动态地加载进来。

这个思路非常重要，因为它告诉我们：**上下文的优化不只是给的更多，而是按需给、分层给、在正确的时机给。**

---

### 第三阶段：Harness Engineering

但是上下文工程其实也不是终点。因为后来大家又发现了一个更麻烦的问题：就算信息给对了，模型也不一定能稳定地执行正确。它可能：

- 计划做的很好，但是执行跑偏了
- 调用了工具，但理解错了返回结果
- 在一个很长的链路里已经慢慢偏航了，但是系统却没有发现

这个时候我们发现啊，提示词和上下文其实主要解决的都是输入侧的问题：提示词优化意图的表达，上下文优化的是信息的供给。但是复杂的任务里还有一个更难的问题：**当模型开始连续行动的时候，谁来监督它、约束它和纠偏它呢？**

这个时候第三阶段来了。Harness 这个词呢，原本的意思是缰绳、马具、约束装置，放到 AI 系统里面，其实就是在提醒我们一件非常朴素的事情：当模型从回答问题走向执行任务，系统不只要负责信息供给，还要能够**驾驭整个过程**。

- 如果前两代工程关注的是怎么让模型**更会想**
- 那 harness 更关注的就是怎么让模型**别跑偏，跑得稳，出了错还能拉回来**

#### 三个概念的通俗对比

假设你要派一个新人去完成一次很重要的客户拜访：

| 概念 | 类比 | 重点 |
|------|------|------|
| **Prompt engineering** | 告诉他任务怎么做："先寒暄，再介绍方案，再问需求，最后确认下一步" | 把话说明白 |
| **Context engineering** | 把资料给齐全：客户背景、过往沟通记录、产品报价、竞品情况、会议目标 | 把信息给对 |
| **Harness engineering** | 让他带着 checklist 去，在关键节点实时汇报，会后核实纪要和录音，发现偏差马上纠正，按明确标准验收结果 | 持续观测、持续纠偏、最终验收 |

所以呢这三者也不是替代的关系，而是**包含的关系**：

- Prompt 是对指令的工程化
- Context 是对输入环境的工程化
- Harness 是对整个运行系统的工程化，它们的边界是一层比一层大

LUNCHA 的工程师给 harness 下了一个很典型的定义：

> **Agent = Model + Harness**
>
> **Harness = Agent - Model**

翻译成人话：在一个 Agent 的系统里面，除了模型本身以外，几乎所有能决定它能不能稳定交付的东西都可以算进 harness。

## 成熟 Harness Engineering 的六层架构

如果拆开来看呢，我自己会把一个成熟的 harness engineering 分成六层：

---

### 第一层：上下文管理（Context Management）

模型能不能稳定发挥，很多时候不仅取决于他聪不聪明，而取决于他看到了什么。所以 harness 的第一职责就是让模型能够在正确的边界内思考。第一层通常包括三件事情：

1. **角色的目标和定义**：模型要知道自己是谁，任务是什么，成功的标准是什么
2. **信息的裁剪和选择**：上下文不是越多越好，而是越相关越好
3. **结构化的组织**：固定的规则放在哪？当前的任务放在哪儿？任务运行的状态放在哪儿？外部的证据又放在哪儿？最好分层清晰

因为信息一旦乱掉呢，模型就很容易漏重点、忘约束，甚至自我污染。

> **💡 具体怎么做（举例）**
>
> **① 角色目标和定义**：在 System Prompt 里明确写清楚身份、目标、成功标准。比如：
> ```
> 你是一个代码审查 Agent。
> 你的任务是：审查用户提交的 PR，找出潜在的安全问题、性能问题、不符合团队规范的写法。
> 成功标准：每条评论必须说明"问题是什么 + 为什么有问题 + 怎么改"，不允许只说"这里有问题"。
> ```
> 这三件事缺一不可——只说"你是审查 Agent"，模型不知道审查什么；只说"找问题"，模型不知道输出格式。
>
> **② 信息裁剪**：不是把所有代码都塞进去，而是只给和当前 PR 相关的文件 diff + 团队规范文档的摘要版，历史 PR 评论只摘最近 3 条。信息太多注意力会分散，模型反而容易漏掉关键问题。
>
> **③ 结构化组织**：把上下文分成四个区域，位置固定：
> - `[系统规则]`：永远不变的约束（写在最前面，利用模型对早期信息的注意力）
> - `[当前任务]`：本次 PR 的目标和范围
> - `[任务状态]`：已审查了哪些文件、发现了几个问题（多轮对话时用）
> - `[外部证据]`：检索到的规范文档片段、工具返回结果
>
> OpenAI 曾经犯过的错误就是把所有规范都写进一个巨大的 `agent.md`，结果 Agent 越来越糊涂。后来改成目录页 + 子文档，Agent 先看索引，按需钻进去——这就是典型的结构化 + 裁剪组合拳。

---

### 第二层：工具系统（Tool System）

没有工具，大模型本质上还是一个文本预测器，会解释会总结，但他接触不到真实的世界。一旦连上工具呢，模型才可以真正的做事，比如搜网页、读文档、写代码、调 API 等等等等。

但是 harness 在这里做的不是简单地把工具挂上去，而是也要解决三个问题：

1. **给他什么工具**：工具太少，能力不够；工具太多，模型又会乱用
2. **什么时候该调用工具**：本来不需要查的时候别乱查，该查的时候也别硬答
3. **工具结果怎么重新喂回模型**：搜索过来的几十条结果不应该原封不动地塞回去，而是要提炼筛选，保持和任务的相关性

> **💡 具体怎么做（举例）**
>
> **① 给什么工具**：Hermes Agent 的 Skills 条件激活机制是个好例子。比如有一个"手动网页搜索"的 Skill，当用户已经配置了 Firecrawl API 的时候，这个 Skill 就自动从索引里隐藏——避免模型在有现成工具的情况下还去走麻烦的 curl+解析流程。原则是：**只在 Agent 的视野里保留它现在能用、该用的工具**。
>
> **② 工具调用时机**：在 System Prompt 里写明工具的调用边界，比如：
> ```
> 如果问题是关于实时数据（股价、天气、最新新闻），必须调用搜索工具，不得凭已有知识直接回答。
> 如果问题是通用概念解释，不需要调用工具，直接回答即可。
> ```
> 没有这个约束，模型会"想当然"地回答实时问题，或者把每个问题都搜一遍浪费 token。
>
> **③ 工具结果处理**：搜索返回了 20 条结果，不应该全部塞回上下文。Harness 层做的是：先过滤掉相关性分数低于阈值的、去掉重复内容、把每条结果压缩成 2-3 句摘要，最后只保留 Top 5 喂回模型。OpenAI 让 Agent 接浏览器时，也不是把整个页面 HTML 都扔进去，而是截图 + 关键文字提取。

---

### 第三层：执行编排（Execution Orchestration）

这一层解决的核心问题就是模型下一步该做什么。很多 agent 的问题呢不是某一步不会，而是不会把所有的步骤给串起来。它会搜索也会总结，也会写代码，但整个过程想到哪做到哪，最后交付出来一堆半成品。

所以一个完整的任务通常需要有这样的轨道：

```
理解目标 → 判断信息够不够（不够继续补）→ 基于结果继续分析 → 生成输出 → 检查输出 → 不满足要求就修正或重试
```

你会发现这已经非常接近人在工作了，区别在于人靠经验，Agent 靠 harness 这套环境。

> **💡 具体怎么做（举例）**
>
> **任务拆解**：让 Agent 在开始执行前先输出一份"执行计划"，比如：
> ```
> 任务：为一个电商 App 写用户评论模块的代码
>
> 执行计划：
> Step 1: 读取现有项目的目录结构和技术栈
> Step 2: 确认评论模块的接口定义（GET /api/reviews, POST /api/reviews）
> Step 3: 编写数据库模型
> Step 4: 编写 API Handler
> Step 5: 编写前端组件
> Step 6: 运行测试验证
> ```
> 有了这个计划，Agent 不会"想到哪做到哪"，也方便在某一步失败时知道从哪里重来。
>
> **Anthropic 的 Planner/Generator 拆分**就是典型的执行编排实践：Planner 先把模糊需求扩展成完整规格，Generator 再逐步实现。两个角色分开，保证"先想清楚再动手"，而不是边想边做越跑越偏。
>
> **循环检查点**：在关键节点强制让 Agent 自问"我现在做的对吗？"比如每写完一个模块，就强制执行一次"是否符合接口定义"的自检，而不是全写完才回头看。这就是 harness 里给 Agent 设的"检查站"。

---

### 第四层：记忆和状态（Memory and State）

没有状态的 agent 呢，每一轮都会像失忆一样，他不知道自己刚做了啥，也不知道哪些结论已经确认了，哪些问题还没解决。所以 harness 还必须要管理状态。

这里至少要让它分清三类东西：

1. **当前任务的状态**
2. **会话中的中间结果**
3. **长期的记忆和用户偏好**

这三类如果混在一起，系统会越来越乱。看清楚之后呢，agent 才会像一个稳定的工作者。

> **💡 具体怎么做（举例）**
>
> **三类状态分开存放，不混用：**
>
> | 类型 | 存哪里 | 示例内容 |
> |------|--------|---------|
> | 当前任务状态 | 对话上下文里的结构化字段 | `{ step: 3, status: "writing_handler", completedFiles: ["model.ts"] }` |
> | 会话中间结果 | 对话历史 or 临时变量 | "已确认接口定义为 POST /api/reviews，参数 userId、content、rating" |
> | 长期记忆/偏好 | 外部数据库 or 文件 | "用户偏好 TypeScript 严格模式；项目用 Prisma ORM；禁止用 any 类型" |
>
> **为什么不能混**：如果把"用户偏好 TypeScript 严格模式"和"当前写到第 3 步了"放在一起，任务结束后清空状态时，长期偏好也丢了；而如果不清空，历次任务的中间状态越堆越多，下次任务启动时模型看到一大堆旧状态，会产生干扰。
>
> **Hermes 的 Memory vs Skill 分工**就是这个思路的实践版本：Memory 存"是什么"（用户偏好、环境细节），Skill 存"怎么做"（可复用流程）——两者物理上就不在同一个地方，不会互相污染。
>
> **Anthropic 的 Context Reset**：当对话轮次太长、上下文快装满时，不是原地压缩，而是启动一个全新的干净 Agent，把当前任务状态（步骤进度、已完成产物）"交接"过去，历史对话全部丢弃。这其实就是在三类状态之间做了精确的拆分——只把"任务状态"带走，其余的清空。

---

### 第五层：评估和观测（Evaluation and Observation）

这个就是很多团队最容易忽视的一层。很多系统其实不是生成不出来，而是生成完了之后根本不知道自己做的好不好。那如果没有独立的评估和观测的能力，agent 就会长期停留在自我感觉良好的状态。

这一层通常包括：

- 输出和验收环境的验证
- 自动的测试
- 日志和指标
- 错误的归因

也就是说，系统不仅要会做，还要知道自己有没有真的做对。

> **💡 具体怎么做（举例）**
>
> **Anthropic 的 Evaluator 独立审查**：Generator 写完代码后，不是让它自己检查，而是交给独立的 Evaluator Agent。Evaluator 不看代码，而是**真实操作页面**——点按钮、填表单、看交互结果，就像一个真实的 QA 测试工程师。这样就避免了"自评失真"——Generator 很难发现自己写的 UI 点击没反应，但 Evaluator 一点就知道了。
>
> **OpenAI 的"让 Agent 看见整个应用"**：
> - 接浏览器：截图 + 模拟用户操作，发现 UI 级别的 bug
> - 接日志系统：查 log 发现运行时错误
> - 接指标系统：查监控看性能数据
>
> 做到这些之后，Agent 不再是"写完就说完成了"，而是有完整的验收链路。
>
> **自动日志 + 错误归因**：每次工具调用失败，不只记录"失败了"，还要记录"为什么失败"——是 API 超时、还是入参格式错、还是模型幻觉了不存在的字段？有归因才能针对性地修，不然下次还会犯同样的错。我自己在书籍对话项目里也用到了类似的思路：SSE 流式输出时，如果中途断连，会把错误类型记录下来，区分是网络问题还是模型返回格式问题，两种情况处理方式完全不同。

---

### 第六层：约束校验与失败恢复（Constraint Validation and Failure Recovery）

最后一层往往才是真正决定这个系统能不能上线的关键环节。因为在真实的环境里面，**失败不是例外，而是常态**。可能搜索不准，可能是 API 超时，也可能文档格式混乱或者模型误解了任务。那如果没有恢复的机制呢，agent 每次出错就只能从头再来。

所以一个成熟的 harness 一定要包括三件事情：

1. **约束**：哪些能做，哪些不能做
2. **校验**：输出之前、输出之后要怎么检查
3. **恢复**：失败之后怎么重试、切入、回滚到稳定的状态

> **💡 具体怎么做（举例）**
>
> **① 约束**：在 System Prompt 里写死"禁止操作"的红线，比如：
> ```
> 禁止：
> - 不允许删除任何数据库记录，只能软删除（设置 deleted_at 字段）
> - 不允许直接修改 main 分支，必须创建 feature 分支
> - 不允许调用外部 API（非白名单内的域名）
> - 生成的 SQL 不允许包含 DROP、TRUNCATE 语句
> ```
> OpenAI 把资深工程师的"分层约束"写成系统规则：哪层不能依赖哪层、什么情况下必须拦截——这不是普通的 lint 规则，而是直接反馈给 Agent 的运行时约束。
>
> **② 校验（输出前 + 输出后）**：
> - 输出前：模型生成 SQL 后，先用规则引擎检查是否包含危险操作，不通过就打回重写
> - 输出后：代码写完后，自动跑单元测试，测试失败就把失败信息 + 应该怎么改一起反馈给模型，进入下一轮修复
>
> OpenAI 的做法就是：规则不只报错，还把**怎么修**也一起告诉 Agent，这样错误信息直接成为下一轮上下文的输入，形成自动修复循环。
>
> **③ 恢复**：失败不是从头来，而是回到上一个已知稳定的检查点。比如：
> - API 超时 → 自动重试 3 次，每次间隔指数增长
> - 模型连续 2 次生成了格式错误的 JSON → 切换到更严格的 prompt 模板重试（而不是无限重试同一个 prompt）
> - 整个任务第 4 步失败 → 回滚到第 3 步完成时的状态，从第 4 步重新执行
>
> 文章开头说的那个真实案例：同样的模型提示词，加了"失败之后怎么恢复"这一层之后，成功率从 70% 拉到了 95% 以上——这就是第六层的核心价值。

## 一线公司的 Harness 实践案例

我们来看最有参考价值的部分——一线公司的真实实践。因为 harness 这个词最近之所以突然火起来呢，不是大家在空谈这个方法论，而是很多公司都已经把它做进了产品和工程体系里面了。

- **LongCha** 在底层模型完全不变的情况下，只通过改造和迭代 Harness，就把他自家的智能体验从榜单排名 30 开外直接杀到了前五
- **OpenAI** 依靠一个只有几名人类工程师的团队，用 agent 从零构建了一个超百万行代码的生产级应用，100% 的代码都是由 agent 编写的，耗时只有纯人工开发的 1/10
- **Anthropic** 也构建了一个可以完全自主编码的系统，只凭一句自然语言的需求，就能在无需人类干预的情况下连续运行几个小时，最后做出完整的游戏、完整的数字音频工作站

---

### Anthropic 的实践

首先啊他们在长程自主任务上总结了两个特别典型的问题。

**问题一："上下文交易"**

时间一长，上下文越来越满，模型就开始丢细节、丢重点，甚至还会出现一种很有意思的现象：他好像知道自己快装不下了，于是开始着急地收尾。

很多系统面对这种问题呢都会做 context compression，也就是把前面的历史上下文压缩一下再继续跑。但 Anthropic 发现对于一些模型来说，这还是不够的，因为压缩只是变短了，不代表那种负担感真的消失了。

所以他们做了一件更激进的事情，叫 **context reset**：不是在原上下文里面继续压，而是换了一个非常干净的新的 agent，把工作交接给他。

> 这个思路很像工程里遇到内存泄漏之后，不是继续清缓存，而是直接**重启整个进程**再恢复状态。这其实就是一种非常典型的 harness 设计。

**问题二：自评失真**

首先模型自己干活，再让他自己给自己打分，往往会偏乐观，尤其是在设计体验、产品完整度这一类没有标准答案的问题上，偏差是更明显的。

所以他们采用了一个非常关键的思路：**把干活的人和验收的人分开**。他们是这样拆分的：

| 角色 | 职责 |
|------|------|
| **Planner** | 负责把模糊的需求扩展成完整的规格 |
| **Generator** | 负责逐步地去实现 |
| **Evaluator** | 负责像 QA 一样去真实地测试 |

更关键的是这个 evaluator 不只是会看代码，而是会真实地操作页面，看具体的交互，检查实际的结果。也就是说这不是一个抽象的审查，它是一个**带具体环境的验证**。

这背后是一个很明确的工程原则：**生产验收必须分离**。只要评估者足够独立，系统就能形成一个真正的有效循环：

```
生成 → 检查 → 修复 → 再检查 → ...
```

---

### OpenAI 的实践

OpenAI 给我的感觉是，他们**重新定义了工程师在 agent 时代的工作**。人类在这个环境里面不需要写一行代码，人类只需要去负责**设计环境**。

具体来说呢，工程师的工作变成了三件事情：

1. 把产品目标拆解成 agent 能理解的小任务
2. agent 失败的时候呢，不是让他更努力一点，而是问**环境里面缺了什么能力**
3. 建立反馈的链路，让 agent 真正地能够看到自己的工作结果

> 当 agent 出了问题的时候，修复方案几乎从来不是要更努力一点，而是确定他缺了什么样的**结构性的能力**。这是典型的 harness 思维。

**实践一：渐进式披露**

他们早期呢犯过一个很多团队都会犯的错误：写了一个巨大的 `agent.md`，把所有的规范、框架、约定全部塞进去了。结果呢 agent 更糊涂了，因为上下文窗口是一个稀缺的资源，塞的太满，其实等于什么都没说。

后来他们的改法：

- 把 `agent.md` 变成一个目录页，只保留核心的索引
- 更详细的内容拆到架构文档、设计文档、执行计划、质量评分、安全规则这些具体的子文档里
- Agent 先看目录，需要的时候再钻进去

你会发现这和我们前面说的 skills 本质上是一个思路：**不是一次性全给，而是按需暴露**。

**实践二：让 agent 看见整个应用**

OpenAI 不只是让 agent 写代码，还会让 agent 看见整个应用。因为产研速度一旦上来呢，瓶颈其实就不再是写，而是验了。那人类根本是验不过来的，所以他们让 agent 自己去验。

怎么验呢？

- **接浏览器**：能截图、点页面，能模拟用户的真实操作
- **接日志系统和指标系统**：让他能够查 log、查监控
- 每个任务都在**独立隔离的环境**在跑，互不影响

结果呢就是 agent 不再是写完代码就说完成了，而是真正地可以跑起来看结果、发现 bug、修 bug、再验证。这其实就是 harness 里非常完整的一套攻击系统：执行编排、评估和观测、约束和恢复。

**实践三：把工程师经验写成系统规则**

OpenAI 不止靠人类在最后的 code review 环节去兜底质量，因为 agent 的提交速度太快了，人类是盯不过来的。所以他们把很多资深工程师的经验直接写成了系统规则，比如：

- 模块怎么分层
- 哪一层不能依赖哪一层
- 什么情况下必须拦截
- 发现问题之后应该怎么修

重点是这些规则不只是负责报错，而是会把**怎么修**也一起反馈给 agent，进入下一轮的上下文。

你会发现，这已经不是传统意义上的代码规范了，而是一套**可持续运行的自动治理系统**。这也是 harness 的典型形态。

---

## 总结

| 阶段 | 解决的问题 |
|------|-----------|
| **Prompt Engineering** | 怎么把任务讲清楚 |
| **Context Engineering** | 怎么把信息给对 |
| **Harness Engineering** | 怎么让模型在真实的执行中持续做对 |

所以 harness 不是在取代 prompt，也不是在取代 context，它是在更大的系统边界上，把前两者都包含进来：

- 当任务还是简单的单轮生成的时候，**prompt** 是很重要的
- 当任务开始依赖外部知识去运行信息的时候呢，**context** 就很关键了
- 当模型真的进入了长链路、可执行、低容错的真实场景里面，**harness** 几乎就是不可避免的

这也是为什么同样的模型在不同的产品里面表现差距会这么大？因为**真正决定上限的可能是模型，但真正决定能不能落地、能不能稳定交付的就是 harness**。

到了这个阶段，我们也看清了一个现实：**AI 落地的核心挑战，正在从"让模型看起来更聪明"转向"让模型在真实世界里稳定地工作"**。如果你最近也在做 agent，我觉得这件事情非常值得你趁早想明白。
