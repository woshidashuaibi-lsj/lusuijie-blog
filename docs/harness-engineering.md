# Harness Engineering（驾驭工程）

> 以 Cursor / CatPaw 等 AI 编程工具为例，深入解析"驾驭工程"的核心思想与实践

---

三个核心维度：

工程师角色转变：从「编写代码」转向「设计环境、明确意图」
Harness Engineering：构建 Agent 专属工具链，实现受控执行
Agent 可读性：北极星指标——代码库重塑为模型能「读懂」的环境

## 一、什么是 Harness Engineering（驾驭工程）

**Harness Engineering**，直译"驾驭工程"，是一种以**工程手段驾驭 AI 能力**的方法论。

在 AI 大模型爆发之前，"工程能力"几乎等同于"写代码的能力"。而现在，LLM 本身已经具备了相当强的编程、推理和分析能力。Harness Engineering 的核心命题是：

> **不是让 AI 替代工程师，而是让工程师学会驾驭 AI —— 通过工程设计让 AI 能力可靠、可控、可组合地为产品服务。**

这个概念在 AI 编程工具领域体现得最为直接。Cursor、CatPaw、GitHub Copilot 等工具的背后，不是把 GPT-4 / Claude 的 API 一接就了事，而是有一整套精心设计的工程体系，让原本"随机性强、幻觉频发"的 LLM，变成一个"稳定、准确、上下文感知"的编程助手。

---

## 二、Harness Engineering 的核心挑战

### 2.1 LLM 的"原罪"

| 挑战                      | 具体表现                                                       |
| ------------------------- | -------------------------------------------------------------- |
| **幻觉（Hallucination）** | 模型自信地写出不存在的函数、错误的 API 签名                    |
| **上下文窗口有限**        | GPT-4 最多 128K tokens，但一个中型项目代码量轻松超过 1M tokens |
| **无状态**                | 每次请求独立，模型不知道"上一次我们讨论过什么"                 |
| **无实时知识**            | 训练截止日期后的新库、新 API 不知道                            |
| **输出不确定**            | 同一个问题，两次回答可能截然不同                               |
| **延迟高**                | 复杂推理需要几秒甚至十几秒，影响用户体验                       |

### 2.2 编程场景的特殊性

- 代码对**精确性**要求极高，一个变量名拼错就报错
- 需要**跨文件理解**：一个函数的实现可能依赖 5 个其他文件的类型定义
- 用户期望**即时响应**：打字停顿 200ms 就要有补全建议
- 代码库是**动态变化**的：每次提交后，AI 理解的"当前代码"就过期了

## 三、核心工程体系

Cursor / CatPaw 等工具通过以下几个核心工程模块来"驾驭"LLM：

```
┌─────────────────────────────────────────────────────────┐
│                    用户输入 / 光标位置                     │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│              上下文工程（Context Engineering）             │
│  代码库索引 → 语义检索 → 上下文裁剪 → Prompt 组装         │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│                  模型路由（Model Routing）                 │
│  轻量补全模型 / 重量推理模型 / 专用代码模型 按需分发        │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│                  输出工程（Output Engineering）            │
│  流式输出 → 结果验证 → diff 渲染 → 安全过滤               │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│                  反馈工程（Feedback Engineering）          │
│  用户接受/拒绝 → 隐式信号收集 → 模型微调/排序优化          │
└─────────────────────────────────────────────────────────┘
```

## 四、上下文工程（Context Engineering）

这是 Harness Engineering 中**最核心、最复杂**的部分。问题本质：**如何在有限的 token 预算内，塞入最有价值的代码上下文？**

### 4.1 代码库索引

工具启动时，会对整个代码库做离线索引：

```
代码文件（.ts / .py / .go ...）
    │
    ▼ AST 解析（Abstract Syntax Tree）
提取：函数签名、类定义、导入关系、注释
    │
    ▼ 向量化（Code Embedding 模型）
每个函数/类/文件 → 语义向量
    │
    ▼ 存储到本地向量索引（如 SQLite + Faiss）
```

**关键点**：代码 Embedding 和普通文本 Embedding 不同，需要专门针对代码训练的模型（如 `text-embedding-3-large` 或 Voyage Code 系列），能理解代码语义而不仅仅是关键词。

### 4.2 实时语义检索

用户在写代码时，工具会**实时**根据当前上下文检索相关代码：

**触发信号**：

- 当前光标所在函数的参数类型
- 正在调用的函数名
- 当前文件的 import 列表
- 用户在 Chat 框里输入的问题

**检索过程**：

```
当前上下文信号
    │
    ▼ 构造查询向量
    │
    ▼ 向量相似度搜索（ANN，< 50ms）
相关代码片段（Top-K）
    │
    ▼ 结合符号图（Symbol Graph）做精确补充
    （如：找到函数调用处，自动也拉入其类型定义）
```

**符号图（Symbol Graph）**：

除了向量检索，还会维护一张代码符号的依赖关系图。当检索到函数 A，会自动拉入 A 的参数类型定义、A 调用的 B 的签名等，确保 LLM 拿到的上下文是**语义完整**的。

### 4.3 上下文裁剪（Context Pruning）

检索到的内容往往远超 token 预算，需要做精细的裁剪：

| 策略             | 说明                                                 |
| ---------------- | ---------------------------------------------------- |
| **相关性排序**   | 按相似度分数降序，优先塞高分片段                     |
| **去重**         | 同一段代码被多次检索到，只保留一份                   |
| **摘要压缩**     | 对长函数只保留签名 + 注释，不保留实现体              |
| **距离优先**     | 同文件 > 同目录 > 同项目，物理距离近的代码优先级更高 |
| **最近修改优先** | git log 最近修改的文件优先级更高                     |

### 4.4 分层上下文设计

Cursor 等工具的 Prompt 通常有多层上下文，按优先级：

```
[System Prompt]
你是一个专业的代码助手，当前工作环境是 TypeScript / React 项目...

[项目级上下文]（从 .cursorrules / .catpaw 等配置文件读取）
项目架构约定、编码规范、禁止使用的 API...

[文件级上下文]
当前文件完整内容（如果不超 token 预算）

[相关代码上下文]（语义检索结果）
Top-K 相关代码片段（带文件路径和行号）

[用户问题 / 当前光标位置]
```

## 五、模型路由（Model Routing）

不同任务对模型能力和速度的需求不同，**用一个模型做所有事**既浪费又慢。

### 5.1 分级路由策略

| 任务类型                  | 特点                                | 适用模型                               |
| ------------------------- | ----------------------------------- | -------------------------------------- |
| **行内补全**（Tab 键）    | 极低延迟（< 100ms），简单上下文推断 | 小型代码专用模型（如 Codestral Mamba） |
| **函数级生成**            | 中等延迟，需要理解类型和接口        | 中型模型（GPT-4o mini / Claude Haiku） |
| **复杂重构 / Agent 任务** | 可接受 5~30s，需要多步推理          | 大型模型（Claude 3.5 Sonnet / GPT-4o） |
| **代码解释 / 文档生成**   | 无延迟要求，文字能力 > 代码能力     | 通用大模型                             |

### 5.2 预测性预加载

当用户打开一个文件时，工具在后台**预先**计算好该文件的上下文向量，并对最可能的下一步操作预热缓存。这样当用户真正触发补全时，检索步骤已经提前完成，端到端延迟大幅降低。

### 5.3 投机采样（Speculative Decoding）

大模型生成 token 很慢，投机采样的思路是：

1. 先用**小模型**快速生成一段草稿（多个 token）
2. 再用**大模型**并行验证这段草稿
3. 验证通过的 token 直接接受，不通过的从大模型重新采样

效果：大模型的**精度** + 小模型的**速度**，实测可以提速 2~3 倍。

## 六、输出工程（Output Engineering）

### 6.1 流式输出（Streaming）

用户不需要等待整个回答生成完毕，LLM 生成一个 token 就立即推送到前端显示。技术上通过 **SSE（Server-Sent Events）** 或 **WebSocket** 实现。

对编程工具来说，流式输出有额外挑战：

- 代码块必须等 ` ``` ` 出现才能开始语法高亮
- diff 视图需要等完整的代码块才能计算行级别的变更
- **解决方案**：推测性渲染，边流边解析代码结构，提前触发高亮

### 6.2 结构化输出与 diff 渲染

当 LLM 修改代码时，工具不是简单地把新代码替换旧代码，而是：

```
LLM 输出的新代码
    │
    ▼ diff 算法（Myers diff / Patience diff）
逐行级别的变更列表（added / removed / unchanged）
    │
    ▼ 渲染为可视化 diff 界面
用户可以逐个 hunk 接受或拒绝
```

这个设计体现了一个重要原则：**AI 提建议，人做决策**。不强制替换，保留用户的控制权。

### 6.3 安全过滤

- **凭证检测**：防止 LLM 在代码里生成真实的 API Key、密码
- **有害代码检测**：检测生成代码中的已知漏洞模式（如 SQL 注入、XSS）
- **版权检测**：检测是否逐字输出了某些开源代码的受版权保护片段

### 6.4 幻觉抑制

专门针对编程场景的输出校验：

- **符号验证**：LLM 生成的代码调用了函数 `foo.bar()`，工具会检查项目中是否真的存在这个方法，如果不存在则给出警告
- **类型验证**：在 TypeScript 项目中，生成代码后在本地跑一次 `tsc --noEmit`，如果报错则提示用户
- **测试运行**：对于修改了已有函数的情况，自动触发相关单测

## 七、Agent 工程（Agentic Engineering）

现代 AI 编程工具已经不满足于单轮问答，而是向**自主 Agent** 进化。

### 7.1 Agent 的工具调用（Tool Use）

Agent 模式下，LLM 不仅输出代码，还可以调用一系列工具：

| 工具              | 能力                                   |
| ----------------- | -------------------------------------- |
| `read_file`       | 读取指定文件内容                       |
| `write_file`      | 写入/修改文件                          |
| `run_terminal`    | 执行终端命令（npm install、pytest 等） |
| `grep_search`     | 在代码库中搜索文本                     |
| `codebase_search` | 语义搜索代码库                         |
| `web_fetch`       | 抓取网页内容（如查文档）               |
| `run_tests`       | 执行测试套件并返回结果                 |

### 7.2 ReAct 循环

Agent 通过 **ReAct（Reasoning + Acting）** 循环解决复杂任务：

```
用户：帮我把所有用到 axios 的地方改成 fetch API

[循环开始]
Thought: 我需要先找出所有 axios 相关的代码
Action: grep_search("import axios")
Observation: 找到 8 个文件，12 处引用

Thought: 需要逐一分析每处用法，再决定替换策略
Action: read_file("src/api/user.ts")
Observation: [文件内容]

Thought: 这里用了 axios.interceptors，需要用 fetch + 手动拦截器替代
Action: write_file("src/api/user.ts", [新内容])
Observation: 写入成功

Thought: 需要验证修改后代码能否编译
Action: run_terminal("npx tsc --noEmit")
Observation: 0 errors

... [继续处理其他文件]

Final Answer: 已完成所有 8 个文件的迁移，TypeScript 编译通过。
```

### 7.3 任务分解与规划

对于复杂任务，现代 Agent 还会先做**规划**，再分步执行：

```
用户：帮我给这个博客加上全文搜索功能

[规划阶段]
1. 分析现有代码架构（read_file × N）
2. 评估技术方案（Algolia / Fuse.js / 自建）
3. 实现索引构建脚本
4. 实现搜索 API
5. 实现前端搜索组件
6. 写测试

[执行阶段]
按规划顺序逐步执行，每步完成后验证
```

### 7.4 Agent 的核心挑战

| 挑战           | 处理方式                                         |
| -------------- | ------------------------------------------------ |
| **任务失控**   | 设置最大工具调用步数，超限后人工介入             |
| **文件误删**   | 每次 write 前做快照，支持一键回滚                |
| **无限循环**   | 检测重复的 Action 序列，自动打断                 |
| **上下文爆炸** | 长任务中定期压缩历史 Action 记录，只保留关键摘要 |

## 八、反馈工程（Feedback Engineering）

### 8.1 隐式信号收集

用户的每个行为都是有价值的训练信号：

| 用户行为                      | 信号含义             |
| ----------------------------- | -------------------- |
| **接受补全**                  | 正样本，这个补全很好 |
| **删除后重写**                | 负样本，这个补全不好 |
| **部分接受**（只保留第一行）  | 弱正样本             |
| **立即 Undo**                 | 强负样本             |
| **Chat 里说"不对，重新生成"** | 负样本               |

### 8.2 A/B 测试框架

工具会对不同的 Prompt 模板、上下文策略、模型版本做线上 A/B 测试：

```
用户群体 ──随机分流──► 策略 A（新 Prompt 模板）
                    └──► 策略 B（旧 Prompt 模板）

指标对比：
- 补全接受率（Acceptance Rate）
- 修改后保留率（Retention Rate after edit）
- 用户满意评分
```

### 8.3 RLHF 与微调

收集足够数据后，可以对基础模型进行微调：

- **SFT（Supervised Fine-Tuning）**：用高质量接受案例做监督微调
- **RLHF（人类反馈强化学习）**：用接受/拒绝信号做偏好优化
- **特定领域适配**：专门针对 TypeScript / Python / Go 等语言做代码偏好优化

## 九、延迟优化工程

AI 编程工具对延迟极度敏感，以下是常见的优化策略：

### 9.1 键入延迟（Typing Latency）优化

```
用户输入字符
    │
    ▼ Debounce（防抖，300ms）
    │ 用户停止输入才触发请求，避免频繁 API 调用
    ▼
预填充上下文（已在后台准备好）
    │
    ▼ 发送 API 请求（流式）
    │
    ▼ 首 token 延迟（TTFT: Time To First Token）
    │ 目标 < 200ms
    ▼
逐 token 渲染
```

### 9.2 缓存策略

| 缓存层                      | 内容                              | TTL            |
| --------------------------- | --------------------------------- | -------------- |
| **Prompt 缓存**（KV Cache） | 相同前缀的 Prompt 在服务端复用 KV | 几分钟         |
| **向量索引缓存**            | 文件未变化时不重新向量化          | 文件修改时失效 |
| **语义搜索缓存**            | 相同查询的检索结果缓存            | 30 秒          |
| **补全结果缓存**            | 光标回到同位置时复用上次结果      | 用户输入时失效 |

### 9.3 并行化

```
用户按下快捷键触发 AI
         │
    ┌────┴────┐
    ▼         ▼
语义检索    当前文件解析
    │         │
    └────┬────┘
         ▼
      组装 Prompt（两路并行）
         │
         ▼
      发送 LLM 请求
```

检索和解析并行执行，节省 50% 的准备时间。

## 十、工程实践中的关键设计原则

### 10.1 优雅降级（Graceful Degradation）

当 AI 服务不可用时（网络超时、服务限流），不能让整个编辑器卡死：

- 补全请求超时 → 静默失败，不弹错误提示
- LLM API 限流 → 自动重试（指数退避） + 降级到更小的模型
- 向量服务不可用 → 退回到关键词搜索

### 10.2 可解释性（Explainability）

AI 改了代码，用户需要知道**为什么**：

- diff 视图标注每处修改的原因
- 引用来源展示（"基于 `src/api/user.ts` 第 42 行的模式"）
- 置信度指示（高置信补全 vs 低置信补全的视觉区别）

### 10.3 隐私与安全

- 用户代码**不应被用于训练**（除非用户明确同意）
- 敏感代码（`.env`、密钥文件）**不发送给 LLM**
- 企业版支持**本地部署模型**，代码不出局域网

### 10.4 人在环路（Human-in-the-Loop）

Harness Engineering 的核心哲学：

> **AI 是副驾驶（Copilot），不是自动驾驶（Autopilot）。**

- Agent 执行有破坏性操作（删文件、跑命令）前**必须确认**
- 所有 AI 修改支持**一键撤销**
- 批量操作前展示预览，而不是直接执行

## 十一、总结：Harness Engineering 的本质

| 传统工程思维   | Harness Engineering 思维         |
| -------------- | -------------------------------- |
| 写代码解决问题 | 设计工程体系让 AI 可靠地解决问题 |
| 关注算法正确性 | 关注 AI 输出的可靠性和边界条件   |
| 优化 CPU/内存  | 优化 token 预算和推理延迟        |
| 测试代码逻辑   | 评估 AI 输出质量（Evals）        |
| 版本控制代码   | 版本控制 Prompt + 模型 + 数据    |

Cursor / CatPaw 等工具的成功，不是因为他们"用了 GPT-4"，而是因为他们在 GPT-4 之上构建了一套精密的**工程体系**：

- 让原本只会聊天的 LLM，变成能看懂整个代码库的编程助手
- 让原本输出不稳定的模型，变成可以在生产环境可靠运行的产品
- 让原本无状态的 API，变成有记忆、有上下文感知的开发伙伴

**这就是"驾驭"的含义：不是被 AI 带着走，而是用工程手段让 AI 为我所用。**
