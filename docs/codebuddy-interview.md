# 腾讯 CodeBuddy AI IDE 团队面试复习

> 基于 Harness Engineering 核心方法论，梳理面试高频考点与答题思路。
> 面试方向：AI 编程工具 / AI IDE / Agent 系统工程

---

## 一、核心概念速查

### 什么是 Harness Engineering？

**一句话定义**：

> 用**工程手段驾驭 AI 能力**的方法论 —— 不是让 AI 替代工程师，而是让工程师学会用工程设计让 AI 可靠、可控、可组合地为产品服务。

**Harness 词义**：缰绳、马具、约束装置 → 引申为"驾驭和约束 AI 的运行系统"。

**核心公式**：

```
Agent = Model + Harness
Harness = Agent - Model
```

即：在 Agent 系统里，除了模型本身以外，所有能决定它是否稳定交付的东西，都算 Harness。

---

### AI 工程三代演进

这是面试最高频的考点之一，需要能流利地讲清三者的区别和演进原因：

| 阶段 | 名称 | 核心问题 | 解决什么 | 局限性 |
|------|------|---------|---------|--------|
| 第一代 | **Prompt Engineering** | 模型有没有"听懂"你说什么？ | 优化意图表达，塑造概率空间 | 只能解决表达问题，解决不了信息缺失 |
| 第二代 | **Context Engineering** | 模型有没有拿到足够且正确的信息？ | 优化信息供给，动态构建上下文 | 只解决输入侧，无法监控和纠偏执行过程 |
| 第三代 | **Harness Engineering** | 模型在真实执行中能不能持续做对？ | 对整个运行系统工程化，驾驭执行过程 | 当前仍在快速演进中 |

**三者关系**：不是替代，而是**包含**（边界一层比一层大）：

```
Prompt ⊂ Context ⊂ Harness
```

**通俗类比**（新员工拜访客户）：

- **Prompt**：告诉他怎么做："先寒暄，再介绍方案，再问需求"
- **Context**：把资料给齐全：客户背景、过往沟通记录、产品报价
- **Harness**：让他带 checklist 去，关键节点实时汇报，会后核实纪要，发现偏差马上纠正，按明确标准验收结果

## 二、AI IDE 的四大核心工程模块

这是 Cursor / CatPaw / CodeBuddy 等工具背后真正的技术体系，面试必考。

```
用户输入 / 光标位置
        │
        ▼
上下文工程（Context Engineering）
代码库索引 → 语义检索 → 上下文裁剪 → Prompt 组装
        │
        ▼
模型路由（Model Routing）
轻量补全模型 / 重量推理模型 / 专用代码模型 按需分发
        │
        ▼
输出工程（Output Engineering）
流式输出 → 结果验证 → diff 渲染 → 安全过滤
        │
        ▼
反馈工程（Feedback Engineering）
用户接受/拒绝 → 隐式信号收集 → 模型微调/排序优化
```

---

### 2.1 上下文工程（Context Engineering）

这是整个 Harness 中**最核心、最复杂**的部分。

**核心命题**：如何在有限的 token 预算内，塞入最有价值的代码上下文？

#### 代码库索引流程

```
代码文件（.ts / .py / .go ...）
    │
    ▼ AST 解析（Abstract Syntax Tree）
提取：函数签名、类定义、导入关系、注释
    │
    ▼ 向量化（Code Embedding 模型）
每个函数 / 类 / 文件 → 语义向量
    │
    ▼ 存储到本地向量索引（如 SQLite + Faiss）
```

**关键点**：代码 Embedding 需要专门针对代码训练的模型（如 `text-embedding-3-large` / Voyage Code 系列），能理解代码语义，而不只是关键词。

#### 实时语义检索

触发信号：
- 当前光标所在函数的参数类型
- 正在调用的函数名
- 当前文件的 import 列表
- 用户在 Chat 框里输入的问题

检索过程：
```
当前上下文信号
    │
    ▼ 构造查询向量
    │
    ▼ 向量相似度搜索（ANN，< 50ms）
相关代码片段（Top-K）
    │
    ▼ 结合符号图（Symbol Graph）做精确补充
    （找到函数 A，自动也拉入 A 的参数类型定义、A 调用的 B 的签名）
```

**符号图（Symbol Graph）**：维护代码符号依赖关系图，确保 LLM 拿到的上下文是**语义完整**的。

#### 上下文裁剪策略

| 策略 | 说明 |
|------|------|
| **相关性排序** | 按相似度分数降序，优先塞高分片段 |
| **去重** | 同一段代码被多次检索到，只保留一份 |
| **摘要压缩** | 对长函数只保留签名 + 注释，不保留实现体 |
| **距离优先** | 同文件 > 同目录 > 同项目，物理距离近的代码优先级更高 |
| **最近修改优先** | git log 最近修改的文件优先级更高 |

#### 分层上下文设计

```
[System Prompt]
你是一个专业的代码助手，当前工作环境是 TypeScript / React 项目...

[项目级上下文]（从 .cursorrules / .catpaw 等配置文件读取）
项目架构约定、编码规范、禁止使用的 API...

[文件级上下文]
当前文件完整内容（如果不超 token 预算）

[相关代码上下文]（语义检索结果）
Top-K 相关代码片段（带文件路径和行号）

[用户问题 / 当前光标位置]
```

#### 渐进式披露（Progressive Disclosure）

这是 Context Engineering 的高级实践，来自 OpenAI 和 Agent Skills 的实践经验：

- **问题**：把十几个工具说明、所有参数定义全塞给模型 → 上下文窗口是稀缺资源，信息一多注意力就涣散
- **解法**：不是一开始把所有能力全给模型看，而是只给最少量的元信息，等他真正要触发某些能力时，再把对应的 SOP、详细参考信息动态加载进来

> **核心原则**：上下文的优化不只是给的更多，而是**按需给、分层给、在正确的时机给**。

---

### 2.2 模型路由（Model Routing）

**核心思想**：用一个模型做所有事，既浪费又慢。

#### 分级路由策略

| 任务类型 | 特点 | 适用模型 |
|---------|------|---------|
| **行内补全**（Tab 键） | 极低延迟（< 100ms），简单上下文推断 | 小型代码专用模型（如 Codestral Mamba） |
| **函数级生成** | 中等延迟，需要理解类型和接口 | 中型模型（GPT-4o mini / Claude Haiku） |
| **复杂重构 / Agent 任务** | 可接受 5~30s，需要多步推理 | 大型模型（Claude 3.5 Sonnet / GPT-4o） |
| **代码解释 / 文档生成** | 无延迟要求，文字能力 > 代码能力 | 通用大模型 |

#### 预测性预加载

当用户打开一个文件时，工具在后台**预先**计算好该文件的上下文向量，并对最可能的下一步操作预热缓存。当用户真正触发补全时，检索步骤已提前完成，端到端延迟大幅降低。

#### 投机采样（Speculative Decoding）

1. 先用**小模型**快速生成一段草稿（多个 token）
2. 再用**大模型**并行验证这段草稿
3. 验证通过的 token 直接接受，不通过的从大模型重新采样

效果：大模型的**精度** + 小模型的**速度**，实测可提速 2~3 倍。

---

### 2.3 输出工程（Output Engineering）

#### 流式输出（Streaming）

通过 **SSE（Server-Sent Events）** 或 **WebSocket** 实现，生成一个 token 就立即推送。

编程工具的额外挑战：
- 代码块必须等 ` ``` ` 出现才能开始语法高亮
- diff 视图需要等完整代码块才能计算行级别变更
- **解决方案**：推测性渲染，边流边解析代码结构，提前触发高亮

#### diff 渲染

```
LLM 输出的新代码
    │
    ▼ diff 算法（Myers diff / Patience diff）
逐行级别的变更列表（added / removed / unchanged）
    │
    ▼ 渲染为可视化 diff 界面
用户可以逐个 hunk 接受或拒绝
```

核心设计原则：**AI 提建议，人做决策**。不强制替换，保留用户的控制权。

#### 安全过滤

- **凭证检测**：防止 LLM 在代码里生成真实的 API Key、密码
- **有害代码检测**：检测已知漏洞模式（如 SQL 注入、XSS）
- **版权检测**：检测是否逐字输出受版权保护的开源代码片段

#### 幻觉抑制

- **符号验证**：检查生成代码调用的函数是否真实存在于项目中
- **类型验证**：TypeScript 项目中生成代码后跑 `tsc --noEmit`，报错则提示
- **测试运行**：修改已有函数后，自动触发相关单测

---

### 2.4 反馈工程（Feedback Engineering）

#### 隐式信号收集

| 用户行为 | 信号含义 |
|---------|---------|
| **接受补全** | 正样本，这个补全很好 |
| **删除后重写** | 负样本，这个补全不好 |
| **部分接受**（只保留第一行） | 弱正样本 |
| **立即 Undo** | 强负样本 |
| **Chat 里说"不对，重新生成"** | 负样本 |

#### A/B 测试框架

```
用户群体 ──随机分流──► 策略 A（新 Prompt 模板）
                    └──► 策略 B（旧 Prompt 模板）

指标对比：
- 补全接受率（Acceptance Rate）
- 修改后保留率（Retention Rate after edit）
- 用户满意评分
```

#### RLHF 与微调

- **SFT（Supervised Fine-Tuning）**：用高质量接受案例做监督微调
- **RLHF（人类反馈强化学习）**：用接受/拒绝信号做偏好优化
- **特定领域适配**：专门针对 TypeScript / Python / Go 等语言做代码偏好优化

## 三、Harness 六层架构（面试重点）

成熟的 Harness Engineering 可以拆成六层，这是**系统设计面试**的核心框架：

### 第一层：上下文管理（Context Management）

核心职责：让模型在正确的边界内思考。

1. **角色的目标和定义**：模型要知道自己是谁、任务是什么、成功标准是什么
2. **信息的裁剪和选择**：上下文不是越多越好，而是越**相关**越好
3. **结构化的组织**：固定规则、当前任务、运行状态、外部证据，分层清晰

> 信息一旦乱掉，模型就很容易漏重点、忘约束，甚至自我污染。

### 第二层：工具系统（Tool System）

没有工具，大模型本质上还是文本预测器。Harness 要解决三个问题：

1. **给什么工具**：工具太少能力不够，工具太多模型又会乱用
2. **什么时候调用工具**：不需要查的时候别乱查，该查的时候别硬答
3. **工具结果怎么重新喂回模型**：搜索到的几十条结果需要提炼筛选，保持与任务的相关性

常见工具清单：

| 工具 | 能力 |
|------|------|
| `read_file` | 读取指定文件内容 |
| `write_file` | 写入 / 修改文件 |
| `run_terminal` | 执行终端命令（npm install、pytest 等） |
| `grep_search` | 在代码库中搜索文本 |
| `codebase_search` | 语义搜索代码库 |
| `web_fetch` | 抓取网页内容（如查文档） |
| `run_tests` | 执行测试套件并返回结果 |

### 第三层：执行编排（Execution Orchestration）

核心问题：模型下一步该做什么？很多 Agent 的问题不是某一步不会，而是不会把所有步骤串起来。

完整的任务轨道：

```
理解目标
    │
    ▼ 判断信息够不够（不够继续补）
    │
    ▼ 基于结果继续分析
    │
    ▼ 生成输出
    │
    ▼ 检查输出
    │
    ▼ 不满足要求就修正或重试
```

**ReAct 循环（Reasoning + Acting）**：

```
用户：帮我把所有用到 axios 的地方改成 fetch API

[循环开始]
Thought: 我需要先找出所有 axios 相关的代码
Action: grep_search("import axios")
Observation: 找到 8 个文件，12 处引用

Thought: 需要逐一分析每处用法，再决定替换策略
Action: read_file("src/api/user.ts")
Observation: [文件内容]

Thought: 这里用了 axios.interceptors，需要用 fetch + 手动拦截器替代
Action: write_file("src/api/user.ts", [新内容])
Observation: 写入成功

Thought: 需要验证修改后代码能否编译
Action: run_terminal("npx tsc --noEmit")
Observation: 0 errors

... [继续处理其他文件]

Final Answer: 已完成所有 8 个文件的迁移，TypeScript 编译通过。
```

### 第四层：记忆和状态（Memory and State）

没有状态的 Agent，每一轮都会"失忆"。需要分清三类：

1. **当前任务的状态**（进行到哪一步了）
2. **会话中的中间结果**（已经确认的结论）
3. **长期的记忆和用户偏好**（跨会话的持久化）

这三类如果混在一起，系统会越来越乱。

### 第五层：评估和观测（Evaluation and Observation）

很多团队最容易忽视的一层。没有评估能力，Agent 会长期停留在"自我感觉良好"的状态。

包括：
- **输出验证**：和验收环境对比
- **自动测试**：单测、E2E 测试
- **日志和指标**：可观测性
- **错误归因**：是模型问题、上下文问题，还是工具问题？

系统不仅要会做，还要知道自己有没有真的做对。

### 第六层：约束校验与失败恢复（Constraint Validation & Failure Recovery）

**核心认知**：在真实环境里，失败不是例外，而是常态。

一个成熟的 Harness 必须包括三件事：

1. **约束**：哪些能做，哪些不能做（合规边界）
2. **校验**：输出之前、输出之后要怎么检查
3. **恢复**：失败之后怎么重试、切换、回滚到稳定状态

## 四、LLM 的核心挑战（背景知识）

面试时体现技术深度，说明为什么需要 Harness：

| 挑战 | 具体表现 |
|------|---------|
| **幻觉（Hallucination）** | 模型自信地写出不存在的函数、错误的 API 签名 |
| **上下文窗口有限** | GPT-4 最多 128K tokens，但中型项目代码量轻松超过 1M tokens |
| **无状态** | 每次请求独立，模型不知道"上一次我们讨论过什么" |
| **无实时知识** | 训练截止日期后的新库、新 API 不知道 |
| **输出不确定** | 同一个问题，两次回答可能截然不同 |
| **延迟高** | 复杂推理需要几秒甚至十几秒，影响用户体验 |

编程场景的特殊性：
- 代码对**精确性**要求极高，一个变量名拼错就报错
- 需要**跨文件理解**：一个函数可能依赖 5 个其他文件的类型定义
- 用户期望**即时响应**：打字停顿 200ms 就要有补全建议
- 代码库是**动态变化**的：每次提交后，AI 理解的"当前代码"就过期了

## 五、延迟优化工程

AI 编程工具对延迟极度敏感，体现工程细节的好机会：

### 键入延迟优化

```
用户输入字符
    │
    ▼ Debounce（防抖，300ms）
    │ 用户停止输入才触发请求，避免频繁 API 调用
    ▼
预填充上下文（已在后台准备好）
    │
    ▼ 发送 API 请求（流式）
    │
    ▼ 首 token 延迟（TTFT: Time To First Token）
    │ 目标 < 200ms
    ▼
逐 token 渲染
```

### 缓存策略

| 缓存层 | 内容 | TTL |
|-------|------|-----|
| **Prompt 缓存**（KV Cache） | 相同前缀的 Prompt 在服务端复用 KV | 几分钟 |
| **向量索引缓存** | 文件未变化时不重新向量化 | 文件修改时失效 |
| **语义搜索缓存** | 相同查询的检索结果缓存 | 30 秒 |
| **补全结果缓存** | 光标回到同位置时复用上次结果 | 用户输入时失效 |

### 并行化

```
用户按下快捷键触发 AI
         │
    ┌────┴────┐
    ▼         ▼
语义检索    当前文件解析（两路并行）
    │         │
    └────┬────┘
         ▼
      组装 Prompt
         │
         ▼
      发送 LLM 请求
```

检索和解析并行执行，节省 50% 的准备时间。

## 六、一线公司实践案例（面试加分项）

### Anthropic 的实践

**问题一：上下文交易（Context Overflow）**

- 现象：上下文越来越满，模型开始丢细节、着急收尾
- 常规方案：Context Compression（压缩历史上下文）
- Anthropic 的做法：**Context Reset** — 换一个干净的新 Agent，把工作交接给他

> 类比：遇到内存泄漏，不是清缓存，而是**重启整个进程再恢复状态**。这就是典型的 Harness 设计。

**问题二：自评失真（Self-evaluation Bias）**

- 现象：让模型自己干活再自己打分，往往偏乐观
- Anthropic 的做法：**把干活的人和验收的人分开**

| 角色 | 职责 |
|------|------|
| **Planner** | 把模糊需求扩展成完整规格 |
| **Generator** | 逐步实现 |
| **Evaluator** | 像 QA 一样真实测试（会真实操作页面、检查交互） |

> 工程原则：**生产与验收必须分离**。只要评估者足够独立，系统就能形成真正有效的循环：`生成 → 检查 → 修复 → 再检查 → ...`

### OpenAI 的实践

**重新定义工程师角色**：人类不需要写一行代码，只需要负责**设计环境**：

1. 把产品目标拆解成 Agent 能理解的小任务
2. Agent 失败时，不是让他更努力，而是问**环境里缺了什么能力**
3. 建立反馈链路，让 Agent 真正看到自己的工作结果

**实践一：渐进式披露**

- 早期错误：写了一个巨大的 `agent.md`，把所有规范、框架、约定全部塞进去 → Agent 更糊涂了
- 改法：`agent.md` 只保留核心索引目录，详细内容拆到子文档，**Agent 先看目录，需要时再钻进去**

**实践二：让 Agent 看见整个应用**

- **接浏览器**：能截图、点页面、模拟用户真实操作
- **接日志系统和指标系统**：能查 log、查监控
- **每个任务在独立隔离的环境跑**：互不影响

结果：Agent 不再是写完代码就说"完成"，而是真正地跑起来看结果、发现 bug、修 bug、再验证。

**实践三：把工程师经验写成系统规则**

- 把资深工程师的经验直接写成系统规则（模块怎么分层、哪层不能依赖哪层、发现问题怎么修）
- 这些规则不只负责报错，还会把**怎么修**反馈给 Agent，进入下一轮上下文
- 这是一套**可持续运行的自动治理系统**，是 Harness 的典型形态

## 七、关键设计原则（面试问"你的技术理念"时用）

### 优雅降级（Graceful Degradation）

当 AI 服务不可用时：
- 补全请求超时 → 静默失败，不弹错误提示
- LLM API 限流 → 自动重试（指数退避）+ 降级到更小的模型
- 向量服务不可用 → 退回到关键词搜索

### 可解释性（Explainability）

AI 改了代码，用户需要知道**为什么**：
- diff 视图标注每处修改的原因
- 引用来源展示（"基于 `src/api/user.ts` 第 42 行的模式"）
- 置信度指示（高置信 vs 低置信补全的视觉区别）

### 人在环路（Human-in-the-Loop）

> **AI 是副驾驶（Copilot），不是自动驾驶（Autopilot）。**

- Agent 执行有破坏性操作（删文件、跑命令）前**必须确认**
- 所有 AI 修改支持**一键撤销**
- 批量操作前展示预览，而不是直接执行

### 隐私与安全

- 用户代码**不应被用于训练**（除非用户明确同意）
- 敏感代码（`.env`、密钥文件）**不发送给 LLM**
- 企业版支持**本地部署模型**，代码不出局域网

## 八、面试高频问题与答题框架

### Q1：介绍一下 Harness Engineering 是什么？

**答题思路**（2 分钟版）：

> Harness Engineering 是一种用工程手段驾驭 AI 能力的方法论。它来源于 AI 工程的三代演进：从 Prompt Engineering（优化意图表达）→ Context Engineering（优化信息供给）→ Harness Engineering（驾驭整个执行过程）。
>
> 核心公式是：`Agent = Model + Harness`。在一个 Agent 系统里，除了模型本身，所有能决定它是否稳定交付的东西都是 Harness，包括上下文管理、工具系统、执行编排、状态记忆、评估观测、约束恢复六层。
>
> 它解决的根本问题是：就算信息给对了，模型也不一定能稳定地连续执行正确。谁来监督它、约束它、在它跑偏时纠偏？这就是 Harness 要做的事。

### Q2：AI IDE 如何解决"代码库太大，上下文窗口不够"的问题？

**答题思路**：

1. **离线索引**：工具启动时对整个代码库做 AST 解析 + Code Embedding，建立本地向量索引
2. **实时语义检索**：用户触发补全时，根据光标位置构造查询向量，< 50ms 内检索到相关代码片段
3. **符号图增强**：检索到函数 A 时，自动拉入 A 的类型定义、A 调用的 B 的签名，确保语义完整
4. **上下文裁剪**：相关性排序、去重、摘要压缩、距离优先，在 token 预算内塞入最有价值的上下文
5. **分层设计**：System Prompt → 项目级上下文 → 文件级上下文 → 检索上下文，优先级明确

### Q3：为什么同样的模型，不同产品效果差距那么大？

**答题思路**：

> 因为真正决定上限的可能是模型，但真正决定能不能落地、能不能稳定交付的是 Harness。
>
> 以我了解到的案例为例：在底层模型完全不变的情况下，只通过改造 Harness，就能把 Agent 任务成功率从 70% 提升到 95% 以上。改动点不是模型，不是提示词，而是：任务怎么拆、状态怎么管、关键步骤怎么校验、失败之后怎么恢复。

### Q4：Agent 长链路任务中，如何处理"上下文越来越长"的问题？

**答题思路**（结合 Anthropic 实践）：

1. **Context Compression**：压缩历史上下文，只保留关键摘要
2. **Context Reset**（Anthropic 实践）：不是在原上下文里继续压，而是启动一个新的干净 Agent，把工作交接过去
3. **状态持久化**：把中间结果结构化存储，而不是全放在上下文里
4. **分步规划**：长任务先做规划，再按规划分步执行，每步有独立的上下文

> 类比：遇到内存泄漏，不是一直清缓存，而是重启进程再恢复状态。

### Q5：如何设计 Agent 的评估体系？

**答题思路**：

核心原则：**生产与验收必须分离**（参考 Anthropic 的 Planner / Generator / Evaluator 角色拆分）

具体措施：
1. **独立 Evaluator**：不让生成模型自评，用另一个模型或规则系统来评估
2. **带环境的验证**：Evaluator 不只看代码，要真实操作页面、检查实际交互结果
3. **自动化测试**：单测、E2E 测试、类型检查（`tsc --noEmit`）
4. **指标体系**：补全接受率、修改后保留率、任务成功率
5. **错误归因**：区分是模型问题、上下文问题、还是工具问题

### Q6：AI IDE 如何处理延迟问题？

**答题思路**（展示工程细节）：

1. **Debounce**：用户停止输入 300ms 后才触发请求，避免频繁 API 调用
2. **预测性预加载**：用户打开文件时后台预热上下文，触发补全时检索已完成
3. **投机采样（Speculative Decoding）**：小模型快速生成草稿，大模型并行验证，提速 2~3 倍
4. **流式输出（Streaming）**：通过 SSE 生成一个 token 就推送一个，TTFT 目标 < 200ms
5. **并行化**：语义检索和文件解析并行执行，节省 50% 准备时间
6. **多层缓存**：Prompt KV Cache、向量索引缓存、语义搜索缓存、补全结果缓存

### Q7：AI 编程工具的核心设计原则是什么？

**答题思路**：

> AI 是副驾驶（Copilot），不是自动驾驶（Autopilot）。核心原则是：
>
> - **AI 提建议，人做决策**：diff 视图让用户逐 hunk 接受或拒绝，不强制替换
> - **人在环路（Human-in-the-Loop）**：破坏性操作必须确认，所有修改支持一键撤销
> - **优雅降级**：AI 服务不可用时静默失败，不让整个编辑器卡死
> - **可解释性**：标注每处修改的原因，展示引用来源
> - **隐私安全**：敏感文件不发给 LLM，企业版支持本地部署

## 九、传统思维 vs Harness Engineering 思维（总结对比）

| 传统工程思维 | Harness Engineering 思维 |
|------------|------------------------|
| 写代码解决问题 | 设计工程体系让 AI 可靠地解决问题 |
| 关注算法正确性 | 关注 AI 输出的可靠性和边界条件 |
| 优化 CPU / 内存 | 优化 token 预算和推理延迟 |
| 测试代码逻辑 | 评估 AI 输出质量（Evals） |
| 版本控制代码 | 版本控制 Prompt + 模型 + 数据 |
| 工程师写代码 | 工程师设计环境，让 Agent 写代码 |
| 靠人力 code review 兜底 | 把工程师经验写成系统规则，自动治理 |
