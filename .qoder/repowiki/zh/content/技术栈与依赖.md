## 引言
本项目 my-blog 是一个基于 Next.js 15.5.3 构建的静态博客系统，采用 TypeScript 提供类型安全保障，结合 Markdown 内容管理与现代化前端工程化流程。本文将深入分析其技术栈构成，重点阐述核心框架、类型系统、样式方案、第三方依赖及构建配置的协同工作机制。

## Next.js 核心框架应用

### 页面路由机制
项目采用 Next.js 的文件系统路由机制，通过 `src/pages` 目录下的文件结构自动生成路由。例如：
- `src/pages/blog/index.tsx` 对应 `/blog` 路由
- `src/pages/blog/[slug].tsx` 实现动态路由，匹配 `/blog/文章标题` 形式的 URL

动态路由 `[slug].tsx` 利用 `getStaticPaths` 预生成所有文章路径，确保静态导出时每个文章页都有对应的 HTML 文件。

### 静态生成（SSG）
项目广泛使用 `getStaticProps` 实现静态生成，提升性能与 SEO 效果：
- 首页 `index.tsx` 获取最新三篇文章作为推荐内容
- 博客列表页 `blog/index.tsx` 获取全部文章数据
- 博客详情页 `[slug].tsx` 根据 slug 加载对应文章内容与元数据
- 照片墙页 `photo/index.tsx` 加载分类照片数据

所有页面均在构建时生成静态 HTML，无需服务器运行时计算。

### 服务器端渲染（SSR）与静态导出
通过 `next.config.js` 中的 `output: 'export'` 配置，整个应用被构建为完全静态的 HTML、CSS 和 JavaScript 文件，适用于部署在 CDN 或对象存储服务上。此模式下，所有 `getStaticProps` 在构建时执行，生成预渲染页面。

## TypeScript 类型安全实践

### 接口定义（src/types/）
项目在 `src/types/` 目录下定义了核心数据结构：
- `blog.ts` 定义 `BlogPost`（文章）、`BlogCategory`（分类）等接口
- `photo.ts` 定义 `Photo`（照片）、`PhotoCategory`（相册分类）等接口

这些接口用于约束数据结构，确保组件接收的数据符合预期。

### 类型推断应用场景
TypeScript 类型推断在以下场景中发挥作用：
- `getStaticProps` 返回的 `props` 对象类型自动推断为组件的 `Props` 类型
- 函数参数如 `getPostBySlug(slug: string)` 明确类型，避免运行时错误
- 组件属性如 `BlogPostPage` 接收 `post: BlogPost` 和 `htmlContent: string`，类型安全确保数据完整性

## CSS Modules 样式隔离机制

### 实现机制
项目采用 CSS Modules 实现组件级样式隔离。每个组件的 CSS 文件命名为 `index.module.css`，如：
- `BlogItem/index.module.css`
- `Comments/index.module.css`
- `Navigation/index.module.css`

在构建时，类名被自动哈希化，避免全局命名冲突。

### 优势
- **作用域隔离**：样式仅作用于当前组件，防止样式泄漏
- **可维护性**：类名无需复杂 BEM 命名，直接使用语义化名称如 `.blogTitle`
- **动态主题支持**：结合 CSS 变量（如 `var(--primary-color)`），便于实现主题切换

## 关键依赖包解析

### @giscus/react
集成 GitHub 评论系统，将 GitHub Discussions 作为评论后端。通过组件化方式嵌入博客文章底部，实现无需后端的评论功能。

### ali-oss
用于与阿里云对象存储（OSS）交互。在 `scripts/upload-oss.js` 中使用，实现构建后自动上传静态资源到 OSS，支持 CDN 加速访问。

### gray-matter
解析 Markdown 文件的 Front Matter 元数据（YAML 格式），提取 `title`、`date`、`tags` 等字段，用于文章列表展示与分类。

### next-mdx-remote
支持 MDX 扩展语法，允许在 Markdown 中嵌入 React 组件。虽然当前项目主要使用 `.md`，但此依赖为未来支持交互式内容提供基础。

### rehype-highlight
在 Markdown 渲染流程中，对代码块进行语法高亮处理。结合 `remark-gfm` 和 `rehype-slug`，提升技术文章的可读性。

## 构建工具链与配置协同

### npm scripts 构建流程
`package.json` 中定义了完整的构建流程：
- `dev`：本地开发
- `build`：构建静态文件
- `export`：别名，实际执行 `next build`
- `deploy`：构建后上传至 OSS，自动化部署
- `upload:images`：上传图片资源
- `upload:blog-assets`：管理博客相关静态资源

### next.config.js 配置解析
```js
const nextConfig = {
  output: 'export',           // 启用静态导出
  trailingSlash: true,        // 所有路径以斜杠结尾
  images: {
    unoptimized: true,        // 禁用图片优化（因静态导出限制）
  },
  pageExtensions: ['js', 'jsx', 'md', 'mdx', 'ts', 'tsx'] // 支持 MDX 扩展
}
```

`output: 'export'` 是静态部署的核心，`unoptimized: true` 表明不使用 Next.js 内置图片优化，适合部署在对象存储的场景。
